# MarkTab Chrome Web Store Release Automation Design

## Goal

Create a repository-owned release pipeline that turns an intentional `vX.Y.Z` Git tag into both a GitHub Release and a Chrome Web Store public-review submission. The pipeline must preserve MarkTab's small, dependency-free extension runtime, reuse the existing validation and packaging commands, and expose failures without claiming that a release succeeded prematurely.

## Scope

This change is limited to release tooling, tests, GitHub Actions configuration, and release documentation.

In scope:

- A GitHub Actions workflow triggered by `v*` tags, with a manual retry entry point.
- Repository-owned Node.js scripts for Chrome Web Store OAuth, upload, status polling, and publication.
- Automated tests that mock all external HTTP traffic.
- Documentation for one-time credential setup and routine releases.

Out of scope:

- Extension runtime code, permissions, data structures, or user-facing behavior.
- Store listing text, screenshots, privacy claims, or permission changes.
- Automatic version mutation inside CI.
- Third-party Chrome Web Store publishing actions or new runtime dependencies.
- Automated rollout-percentage management after review approval.

## Chosen Approach

Use GitHub Actions only as the orchestrator and keep Chrome Web Store API behavior in focused, repository-owned Node.js modules. This is preferred over inline YAML and `curl` because the API behavior can be tested, retried, and understood independently. It is preferred over a third-party publishing action because it keeps the credential and supply-chain surface small.

The automation will be developed on `codex/chrome-store-release-automation`, based on `codex/newtab-ui-layout`. Its pull request will initially target the UI branch so the review contains only automation changes. After PR #3 merges, the automation pull request will be retargeted to `main`.

## Release Contract

- A pushed tag matching `v*` is an intentional production release request.
- The canonical release version is the tag without its leading `v`.
- The tag version must exactly match both `package.json` and `manifest.json`.
- CI never edits or commits a version. Maintainers use the existing `npm run bump -- X.Y.Z` command before creating the tag.
- The release artifact is the existing `dist/marktab-X.Y.Z.zip` generated by `npm run package`.
- The Chrome Web Store publication target is the public/default channel, which submits the uploaded version for review.

## Workflow Architecture

Add `.github/workflows/release.yml` with two entry points:

- `push.tags: ['v*']` for normal releases.
- `workflow_dispatch` with a required tag input for an operator-controlled retry.

The workflow performs these stages in order:

1. Resolve the requested tag and check out that exact tagged commit.
2. Validate the tag format and assert version equality across the tag, `package.json`, and `manifest.json`.
3. Run `npm test`, `npm run validate`, and `npm run package`.
4. Create or reuse a draft GitHub Release for the tag and upload the ZIP, replacing an existing same-named asset on retry.
5. Exchange the stored OAuth refresh token for a short-lived access token.
6. Upload the ZIP to the Chrome Web Store Publish API.
7. Poll the item status until upload processing reports success or a bounded timeout/error occurs.
8. Submit the successful draft to the public/default publication target.
9. Mark the GitHub Release as published only after the Chrome Web Store accepts the publication request.

The workflow uses a repository-wide Chrome Web Store concurrency group with cancellation disabled. This serializes releases so two tags cannot race to update the same store item.

## Components

### Chrome Web Store API client

A focused module owns HTTP operations:

- Exchange refresh credentials at Google's OAuth token endpoint.
- Upload a ZIP to the Chrome Web Store item upload endpoint.
- Read upload status while processing is in progress.
- Submit the item to the public/default publish endpoint.
- Normalize non-2xx responses, API error payloads, and unexpected response bodies into safe errors.

The module accepts injected `fetch` and timing functions where useful so tests can run without network access or real delays.

### Publication command

A small command-line script reads the artifact path and required environment variables, invokes the API client in sequence, and writes concise status messages suitable for Actions logs. It validates inputs before making a network request and never logs secret values or authorization headers.

### Local OAuth bootstrap helper

`npm run cws:auth` starts a temporary loopback listener on `127.0.0.1`, opens Google's authorization page, receives the OAuth callback, and exchanges the authorization code for a refresh token using the `https://www.googleapis.com/auth/chromewebstore` scope.

The helper reads the client ID and client secret from the local environment. When GitHub CLI is available and authenticated, it pipes the resulting refresh token directly to `gh secret set CWS_REFRESH_TOKEN` through standard input. The refresh token, client secret, and access token are not printed or written to the repository.

## Credentials And Permissions

The repository requires four Actions secrets:

- `CWS_CLIENT_ID`: OAuth desktop-client ID.
- `CWS_CLIENT_SECRET`: OAuth desktop-client secret.
- `CWS_REFRESH_TOKEN`: long-lived refresh token generated by the bootstrap helper.
- `CWS_ITEM_ID`: the already-published MarkTab Chrome Web Store item ID.

The workflow receives only `contents: write`, which is required to create and publish GitHub Releases. It does not run on pull requests, so untrusted pull-request code cannot receive release secrets. Secrets are passed only to the Chrome Web Store publication step, not to tests, validation, or packaging.

Any credential pasted into chat or another uncontrolled channel must be rotated before use. No credential value is committed to the repository, test fixture, documentation, or workflow file.

## Failure Handling And Retry

- A malformed tag or version mismatch fails before packaging or external writes.
- Test, validation, or packaging failure prevents creation of a release or store upload.
- A GitHub Release remains a draft until the store submission is accepted.
- OAuth, upload, processing, or publish failures stop the workflow with a sanitized error and an Actions step summary.
- Upload polling uses a fixed maximum duration and fails rather than waiting indefinitely.
- The manual workflow checks out the requested existing tag and repeats the same deterministic artifact build.
- On retry, the GitHub Release is reused and its same-named ZIP asset is replaced.
- Every retry submits the same deterministically rebuilt ZIP. If that upload response is already successful, the command proceeds directly to publication; if processing is still in progress, it polls before publishing.
- If the API refuses a repeated upload because the remote item has advanced beyond a safely resumable state, the workflow reports that state and requires dashboard inspection.
- If the store reports a terminal state that cannot be resumed automatically, the workflow stops and reports the state; it does not guess, mutate the version, or publish the GitHub Release.

The pipeline is idempotent where the external APIs expose enough state to prove the next action is safe. It deliberately fails closed for ambiguous store states.

## Testing

Use Node's built-in test runner and no new test dependency.

Automated tests cover:

- Successful OAuth token exchange, upload, processing poll, and publish request.
- OAuth rejection and missing-token responses.
- Upload rejection, upload processing failure, timeout, and malformed payloads.
- Publish rejection and safe error redaction.
- Required environment variables and missing artifact validation.
- Tag/package/manifest version agreement and mismatch behavior.
- Static workflow contracts: tag trigger, manual dispatch, serialized concurrency, minimal permissions, existing quality commands, draft-first release ordering, and secret names.

Before completion, run:

- `npm test`
- `npm run validate`
- `npm run package`

No automated test contacts Google, GitHub, or the real Chrome Web Store item. The first production tag remains the integration proof, and its GitHub Actions log and Chrome Web Store Developer Dashboard state must both be inspected.

## Documentation

Update `README.md` and `CHROME_STORE_SUBMISSION.md` to describe:

- One-time Google Cloud OAuth and GitHub Secrets setup.
- Refresh-token generation without exposing credentials.
- The normal version bump, commit, tag, and push sequence.
- What the workflow publishes automatically.
- How to retry an existing tag and interpret a draft GitHub Release after failure.
- The remaining manual responsibility: monitor and respond to Chrome Web Store review results.

## Acceptance Criteria

- Pushing a valid `vX.Y.Z` tag builds the existing release ZIP, uploads it to a draft GitHub Release, uploads it to the existing Chrome Web Store item, and requests public review.
- The GitHub Release becomes public only after the store API accepts the publication request.
- Version mismatch, failed quality checks, missing secrets, API failure, and processing timeout all fail clearly without publishing a misleading GitHub Release.
- A maintainer can manually retry an existing tag without creating a new version.
- Store credentials never appear in committed files or normal logs.
- The workflow uses no third-party Chrome Web Store action and introduces no extension runtime dependency or permission.
- Existing `npm test`, `npm run validate`, and `npm run package` checks pass.
